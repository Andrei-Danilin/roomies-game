# Code Patterns & Conventions — chaoshappens.com

## Language conventions
- One component per homepage section under `components/sections/` (`Header.tsx`, `Hero.tsx`, `Status.tsx`, `About.tsx`, `Needs.tsx`, `Chars.tsx`, `HowTo.tsx`, `Gallery.tsx`, `Media.tsx`, `Notify.tsx`, `Contact.tsx`, `Faq.tsx`, `Footer.tsx`), each taking `{ content: Content }` (from `lib/content.ts`) as its prop and rendering that locale's copy — no hardcoded strings, no hooks of their own.
- Interactive fragments (`components/NotifyForm.tsx`, `components/FaqAccordion.tsx`, `components/LanguageSwitcher.tsx`) are the only components with their own `'use client'` + local state; they take a narrower prop (e.g. `Content['notify']`) rather than the whole `Content` tree.
- Design tokens (fonts, shared colors) live in `lib/theme.ts` (`fontDisplay`, `fontBody`, `colors.*`) — import these instead of repeating string literals like `'Fredoka, sans-serif'` across section files. Per-section accent colors that appear once (e.g. a section's kicker color) stay inline; only repeated tokens belong in `lib/theme.ts`.
- `needsMeta` / `charsMeta` (icons, brand colors, stable keys) stay in `lib/content.ts`, never in `content/<locale>.json` — components merge them with per-locale copy by `key` at render time (see Needs.tsx, Chars.tsx).

## Client/server boundary
- Server Components by default. `app/page.tsx` is the one exception at the page level — it's `'use client'` because it reads the visitor's locale via `useContent()` (`lib/locale-context.tsx`) and needs to re-render all sections on language switch without a full reload. See ADR-006 in `decisions.md` for why, and its tradeoffs.
- `lib/locale-context.tsx`'s `LocaleProvider` hydrates the persisted locale via `useSyncExternalStore` (not `useEffect` + `setState`) — this avoids both an SSR/CSR text mismatch and the `react-hooks/set-state-in-effect` lint rule. It also syncs `document.documentElement.lang` in a `useEffect`, which is legitimate there since it's synchronizing React state with a DOM attribute outside React's tree, not calling `setState`.

## Error handling pattern
- `/api/notify` (`app/api/notify/route.ts`) validates the email independently of the client (never trusts `NotifyForm`'s client-side regex as the real boundary check), then calls `lib/mailerlite.ts`'s `subscribeToWaitlist`. MailerLite failures throw a typed `MailerLiteError` (message + status) that the Route Handler catches and maps to a generic client-facing error message (`502`) — the specific MailerLite error detail (which could reveal account/API internals) never reaches the browser, only server logs. Any *other* thrown error (a bug, not a MailerLite-shaped failure) is rethrown rather than swallowed, so it surfaces as a real 500 instead of being silently miscategorized as a MailerLite problem.
- MailerLite's own API treats "subscriber already on the list" as a success (200, not an error) — `subscribeToWaitlist` doesn't special-case duplicates.
- `/api/export-subscribers` follows the same shape: typed errors (`MailerLiteError`, `GoogleSheetsError`) from the `lib/` clients, caught in the Route Handler, mapped to a generic response — but since it's a cron job with no user-facing surface, the response body doesn't need a friendly message, just `console.error`-logging the real cause for Vercel's function logs before returning a 502.

## Logging pattern
Route Handlers with no user-facing surface (`/api/export-subscribers`, a cron job — nobody sees its response) `console.error` the real failure cause before returning a generic error status, so Vercel's function logs have something to look at. Handlers with a client surface (`/api/notify`) do the same but additionally keep the response body generic so internals never leak to the browser (see the Error handling pattern above). No structured logging library — plain `console.error` is sufficient for this project's traffic/log volume.

## Cron job pattern
- `/api/export-subscribers` is triggered by Vercel Cron (`vercel.json`'s `crons` array, weekly). It authenticates the request via `CRON_SECRET`: Vercel automatically sends `Authorization: Bearer ${CRON_SECRET}` for cron-triggered requests, so the handler just compares that header — no other auth mechanism needed, and the route can't be invoked by a guessed public URL without the secret.
- The handler stays thin (fetch MailerLite subscribers + read existing Sheet rows in parallel via `Promise.all`, diff, append) — the actual "what counts as new" decision lives in `lib/subscriber-export.ts`'s `findNewSubscribers`, a pure function with no I/O, kept separate specifically so it's unit-testable without mocking HTTP.
- Cron jobs should be idempotent/reconciliation-based (Vercel's own guidance: delivery is best-effort and can occasionally double-invoke). This job re-reads the Sheet's existing emails at the start, then re-reads and re-filters again immediately before appending — that second check shrinks the window where two overlapping invocations could both append the same new subscriber, but doesn't fully eliminate it (a true guarantee would need a distributed lock, e.g. Redis, which isn't worth the added infra for a low-traffic weekly job at this stage). A duplicated row is a nuisance, not data corruption — worth revisiting if the waitlist grows enough for overlapping runs to become a real risk.

## Google Sheets service-account auth pattern
- `lib/google-sheets.ts` authenticates server-to-server via a Google service account JWT (RS256-signed with `crypto.createSign`, exchanged for an OAuth2 access token at `oauth2.googleapis.com/token`) — no `googleapis` npm dependency, matching `lib/mailerlite.ts`'s existing thin-fetch-client convention. This is the right auth flow for a non-interactive weekly job (no user present to consent), unlike the interactive GitHub OAuth popup flow in `app/api/auth`.
- `GOOGLE_SERVICE_ACCOUNT_PRIVATE_KEY` is stored with literal `\n` escapes (Vercel env vars can't hold real newlines cleanly) — `lib/google-sheets.ts` unescapes them (`.replace(/\\n/g, '\n')`) before passing the PEM to `crypto.createSign`.
- The Sheet is a running history, not a snapshot: `readExistingEmails` reads what's already there, `appendRows` only adds the diff — a run is never destructive to previously-recorded rows, even if `content/<locale>.json`-style overwrite-on-each-run would have been simpler to implement.

## Content-loading pattern
`lib/content.ts` statically imports all three `content/<locale>.json` files and exposes `getContent(locale)`, which falls back to `defaultLocale` ('en') for an unrecognized locale string. `needsMeta`/`charsMeta` are exported alongside as code-owned constants; components join them to a locale's `content.needs.items[key]` / `content.chars.items[key]` by the shared `key`.

## Decap CMS / OAuth proxy pattern
- `public/admin/config.yml`'s field tree must mirror `Content` (`lib/content.ts`) exactly — one `files` entry per locale (`content/en.json`, `es.json`, `ru.json`) sharing a single field-tree via a YAML anchor (`fields: &content_fields`) so the three locales can't drift out of sync. Only the copy tree is exposed; `needsMeta`/`charsMeta` never appear in `config.yml` (per ADR-005).
- Decap CMS's flow-mapping (`{ label: X, name: y, widget: z }`) and block-style YAML don't mix — a flow mapping can't contain a nested block sequence. Nested `object`/`list` fields must use block style (`fields:` on its own line, `-` items indented below) all the way down; only leaf field definitions (no `fields:` of their own) can use the compact `{ ... }` flow form. Validate with `node -e "require('js-yaml').load(require('fs').readFileSync('public/admin/config.yml','utf8'))"` before trusting it.
- `next.config.ts` has a `rewrites()` entry mapping `/admin` (and `/admin/`) → `/admin/index.html`, because Next's static file serving under `public/` doesn't auto-resolve a directory's `index.html` the way Netlify/Apache do. Because of that rewrite, the browser's URL bar shows `/admin` (no trailing slash) even though the served file lives at `/admin/index.html` — Decap's default relative `config.yml` fetch would then resolve against `/` and 404. `public/admin/index.html` works around this with an explicit `<link rel="cms-config-url" href="/admin/config.yml">`.
- The GitHub OAuth handshake (`app/api/auth/route.ts` + `app/api/auth/callback/route.ts`) implements the same popup-window `postMessage` protocol as `netlify-cms-oauth-provider`: the callback page posts `"authorizing:<provider>"` to `window.opener`, waits for any reply, then posts `"authorization:<provider>:success:{...}"` (or `:error:{...}`) — this exact message shape is what Decap's `github` backend listens for, not something invented per-project.
- CSRF protection for the OAuth `state` param uses a short-lived httpOnly cookie (`decap_oauth_state`, scoped to `/api/auth` so it's sent on the `/api/auth/callback` sub-path) set in `/api/auth` and checked in `/api/auth/callback` — not a server-side session store, since Vercel Route Handlers are stateless between requests. The cookie's `secure` flag is tied to `NODE_ENV === 'production'`, not hardcoded `true` — plain HTTP local dev (`npm run dev`) would otherwise silently drop it.
- `app/api/auth/handshake.ts` holds the shared `renderHandshakePage`/`htmlResponse` helpers used by both `/api/auth` and `/api/auth/callback`, so every failure mode (missing env var, bad state, GitHub error, network failure) renders the same popup handshake page instead of some paths returning raw JSON that the popup-window flow can't do anything with. The handshake script only relays the token/error to `window.opener` if the replying window's origin matches `window.location.origin` — the popup and the admin page that opened it are always same-origin in this flow, so anything else is untrusted and ignored.
- The GitHub token-exchange `fetch()` + `.json()` in `callback/route.ts` is wrapped in `try/catch`: an uncaught network failure or non-JSON response would otherwise propagate out of the Route Handler, and Next would serve its generic error page instead of the handshake page the popup is waiting on — leaving the CMS login hung with no visible error.
- Decap CMS itself is loaded from a CDN (`public/admin/index.html`) rather than an npm dependency, pinned to an exact version (not a `^` range) since it's unreviewed code running with git-push credentials — bump it deliberately, as a reviewed change to that one line.

## Testing pattern
- Unit (vitest + Testing Library): render a component/hook and assert on visible text/roles — see `lib/content.test.ts` (pure data), `lib/locale-context.test.tsx` (hook + persistence via a small probe component), `app/page.test.tsx` (full page, wrapped in `<LocaleProvider>` since `Home` now needs that context).
- `vitest.setup.ts` must call `cleanup()` in an `afterEach` — this project's `vitest.config.ts` doesn't set `globals: true`, so Testing Library's automatic cleanup doesn't self-register; without it, multiple tests in one file leak DOM nodes into each other.
- E2E (Playwright): `e2e/home.spec.ts` (page loads, key content visible), `e2e/language-switcher.spec.ts` (switching locale updates visible copy and persists across reload, for each non-default locale plus the no-preference default). Playwright's own dev server runs on port 3100 (`playwright.config.ts`) to avoid colliding with anything already on 3000/3001.
